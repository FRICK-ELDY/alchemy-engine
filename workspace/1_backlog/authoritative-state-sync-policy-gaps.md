## ギャップ一覧

### G1 — 権威サーバー tick がポリシーと不一致（設定不能）

**ポリシー**: Elixir 公式 tick を **10 / 20 / 30 Hz**（用途別）。60 Hz 近傍は非推奨。

**現状**: `Contents.Events.Game` が `@tick_ms 16`（約 **62.5 Hz**）で `Process.send_after` により駆動している（`apps/contents/lib/events/game.ex`）。診断モジュールも `@tick_ms 16` を前提（`apps/contents/lib/events/game/diagnostics.ex`）。

**ギャップ**:

- 10 / 20 / 30 Hz の **設定スキーム**（`config` またはルーム単位）がない。
- デフォルトがポリシーの「推奨 20Hz」や「高速 30Hz」と一致していない。
- 負荷試験・レポート用の **10Hz プロファイル** の切替がコード上明示されていない。

---

### G2 — 「公式に適用される入力列」の明示的モデルがない

**ポリシー**: 採用順・採否は **サーバーが宣言**。Zenoh 等で Rust 経路があっても **二重の真実** を作らない。

**現状**:

- WebSocket: `Network.Channel` が `send(pid, {:move_input, ...})` でルームプロセスへ直送。
- Zenoh: `Network.ZenohBridge` が protobuf 解釈後、同様に `Contents.Events.Game` へメッセージ配送（`apps/network/lib/network/zenoh_bridge.ex`）。
- マルチクライアント時の **全入力の全順序**（同一 tick 内の合流、遅延・再送）を表す **データ構造・API・ドキュメント**は未整備。

**ギャップ**:

- 「公式入力ログ」相当の **キュー／シーケンス番号／タイムスタンプ規約**。
- 複数トランスポート（Channel / UDP / Zenoh）から同一ルームに入った入力の **単一パイプへの正規化**。
- 不正入力の **サーバー側検証**ポリシーと実装フック（レート制限、スキーマ検証の統一）。

---

### G3 — Rust 高頻度シミュレーションと Elixir「確定フレーム」コミットのパイプがない

**ポリシー**: Rust を 60Hz 等で回す場合、**いつ・どの形式で Elixir が取り込みコミットするか** をプロトコルで固定。

**現状**: [overview.md](../../docs/architecture/overview.md) どおりサーバー NIF は **Formula VM のみ**で、旧 60Hz 物理ループは撤去済み。Elixir contents がシーン駆動の主担当。

**ギャップ**:

- サーバー側 Rust で **補助シム**を回す場合の **境界プロトコル**（コミット境界、ロールバック有無）。
- Elixir の tick（G1 調整後）と Rust サブステップの **整合アルゴリズム**の未設計・未実装。

---

### G4 — クライアント描画 FPS（ユーザー設定・自動デフォルト・HMD 上限）

**ポリシー**: ユーザー設定、デフォルト 30/60（スペック連動）、上限はモニタ／**HMD リフレッシュ**に合わせる。

**現状**:

- Rust レンダラは **FPS 計測表示**に近い用途（`rust/client/render` の `current_fps` 等）。**上限固定・ユーザー設定・設定永続化**は未対応とみなす。
- `client_info.proto` は `os` / `arch` / `family` のみ（`proto/client_info.proto`）。**GPU 段階・ディスプレイ／HMD リフレッシュ**の報告がない。
- ZenohBridge は `client_info` を ETS に保存するが、ポリシーに必要なフィールドがプロトコルにない。

**ギャップ**:

- 設定 UI／設定ファイルとレンダラ（vsync、フレームペーサー）の接続。
- スペック検出と **推奨プリセット**（30 vs 60）のロジック。
- OpenXR 利用時の **表示モード**とデスクトップウィンドウの扱いの整理。

---

### G5 — ネットワーク品質の「アプリ実測」（帯域・損失・RTT）の集約

**ポリシー**: 契約 Mbps だけでなく **送受バイト／秒・損失・RTT** を取得し、品質プリセットや UI に反映。

**現状**: Phoenix Channel に `ping` / `pong` がある（`Network.Channel`）。Zenoh 経路では同等の **統一メトリクスパイプ**は見当たらない。

**ギャップ**:

- クライアント／サーバー双方での **メトリクスサンプリング**と Telemetry またはログ基盤への送出。
- 「品質プリセット」に結びつく **閾値設計**と UI 表示。
- UDP / WebSocket / Zenoh の **横断比較**可能な単位の定義。

---

### G6 — インタレスト管理（距離カット・同時可視上限）

**ポリシー**: クライアント調整可能、デフォルトは **ハードウェア自動推奨**。サーバー側ハードキャップは将来検討。

**現状**: `render_frame.proto` は **シーンの描画命令列**中心で、**購読半径・最大同時エンティティ・LOD 方針**を表すフィールドはない（サーバーが全クライアントに同一フレームを送る前提に近い）。

**ギャップ**:

- プロトコル（または別トピック）での **可視性パラメータ**のやり取り。
- サーバー側の **フィルタリング**（帯域削減）とクライアント側 **詳細度**の分担設計。
- ポリシーどおりの **不正対策用キャップ**のオプション設計。

---

### G7 — ロール（RBAC）のサーバー強制

**ポリシー**: `admin` > `builder` > `moderator` > `guest` > `spectator`。**サーバーで毎回検証**。

**現状**: ルーム参加は `Network.RoomToken` 等があるが、上記ロール階層と **操作ごとの authorize** はコードベースに体系化されていない（コンテンツごとの ad hoc のみ）。

**ギャップ**:

- ロールの **データモデル**（ユーザー・ルーム・コンテンツ単位）。
- `join` / 入力 / ワールド編集 / モデレーション API への **共通ガード**。
- 監査ログ（誰がいつ何をしたか）の要否と保存先。

---

### G8 — リモートコンテンツ: HTTPS ＋ 署名の一貫実装

**ポリシー**: 当面 **HTTPS ＋ 署名**（マニフェスト／ハッシュと作者または Hub の署名）。

**現状**: [asset-cdn-design.md](./asset-cdn-design.md) に `https://` とパッケージ化の方向性があるが、ポリシー用語（**署名検証・鍵ローテーション・マニフェスト形式**）との **要件トレーサビリティ**は別作業。実装は部分的。

**ギャップ**:

- ダウンロードパイプラインでの **署名検証**（Rust `AssetLoader` 経路含む）。
- マニフェスト仕様（バージョン、依存、失効）の確定と実装。
- Hub／インスタンス間で共有可能な **信頼の鎖**ドキュメントとテスト。

---

### G9 — 同時接続・公式サポート数のベンチマークとレポート

**ポリシー**: ホスト依存を認めつつ、**ベンチマークとレポート**のうえで公式宣言。参考目標として大人数＋興味管理の文脈。

**現状**: `Core.StressMonitor` 等に断片的な監視はあるが、ポリシーに沿った **再現手順・環境表・SLO 表**のレポート生成は未整備。`scale-and-gaps.md` のチェックリストと重複する作業あり。

**ギャップ**:

- シナリオ定義（人数・tick・プロトコル・クライアント数）。
- 自動化ベンチと **人間可読レポート**（HTML/Markdown）の出力。
- 「公式サポート帯」の **ドキュメント上の宣言プロセス**。

---

### G10 — ドキュメントとコードの文言齟齬

**ポリシー / overview** と矛盾しうる **モジュールドキュメント**の更新が必要。

**例**: `Network` の `@moduledoc` に「並行 **60Hz** 物理演算を実証」等の記述が残っている（`apps/network/lib/network.ex`）。現行アーキテクチャ（overview）および Elixir tick 方針（ポリシー）と揃えると誤解が減る。

**ギャップ**: 該当ドキュメントの棚卸しと修正タスク。

---

## 次のアクション（提案）

1. **G1** を `2_todo` に落とす（設定化とデフォルト 20Hz への合わせ方を決める）。
2. **G2** と **G3** は設計タスクとして先に **シーケンス図＋プロトコル草案** を切ると、実装が並行しやすい。
3. **G4〜G6** はクライアント体験とスケールの両方に効くため、`client_info` / 設定 / メトリクスを **一枚の「クライアント能力プロトコル」** にまとめる案を検討。
