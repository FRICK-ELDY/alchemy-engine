# 新コンテンツ: Rolling Ball Maze（ローリングボール迷路）

> 作成日: 2026-03-02

---

## 目的

### エンジン観点

`BulletHell3D` が「既存の2D ECS を3Dに転用する」課題を探るのに対し、
本コンテンツは **Rust 側に3D物理（重力・転がり・傾斜面衝突）が存在しない** という
より根本的な課題に向き合うことを目的とする。

具体的に明らかにしたい課題：

| 課題カテゴリ | 確認したい問い |
|:---|:---|
| 3D 物理の不在 | 重力・転がり・傾斜面衝突を Elixir 側で近似するとどこまで自然に見えるか？ |
| メッシュ形状の限界 | 現在の `Box3D` のみで「球」「傾斜面」「穴」を表現できるか？ |
| DrawCommand 設計 | `Box3D` / `GridPlane` / `Skybox` 以外の DrawCommand が必要になるか？ |
| Elixir 側物理近似 | Elixir で毎フレーム物理計算をするとフレームレートに影響するか？ |
| シーン遷移 | 「落下」「クリア」という状態遷移を既存の SceneManager で自然に表現できるか？ |

### コンテンツ観点

傾けた台の上でボールを転がし、穴に落ちないようにゴールを目指す。
「3D ならではのゲームプレイ」を最もシンプルな形で実現する。

---

## 何を作るか

### ゲーム概要

| 項目 | 内容 |
|:---|:---|
| ジャンル | 3D バランスボール（ローリングボール迷路） |
| 視点 | 固定俯瞰カメラ（斜め上から見下ろし） |
| 操作 | WASD でボールに力を加える（加速度制御） |
| 目標 | ゴールエリアにボールを転がして到達する |
| 失敗条件 | 穴に落ちる（Y 座標が一定以下になる） |

### ゲームプレイ

```
[ボール（プレイヤー）]
  - 白いボックス（球の代替）
  - WASD で XZ 方向に加速度を加える
  - 慣性あり（速度が徐々に減衰する）
  - フロアの穴に落ちると失敗

[フロア]
  - グリッドで構成された平面
  - 一部のグリッドに「穴」がある（ボールが通過すると落下）

[ゴール]
  - 緑色のボックスで示されたエリア
  - ボールが接触するとクリア

[障害物]
  - 赤いボックスが配置されており、ボールが当たると弾かれる
```

### ステージ構成

ステージは 3 面。各面でフロアの形状・穴の配置・障害物が変わる。

| ステージ | フロアサイズ | 穴の数 | 障害物 |
|:---|:---|:---|:---|
| 1 | 10×10 | 4 | なし |
| 2 | 12×12 | 8 | 2 個 |
| 3 | 15×15 | 16 | 4 個（動く） |

### シーン構成

```
Title
  │
  └──→  Playing（ステージ 1）
            │
            ├── ゴール到達 ──→  StageClear ──→  Playing（次ステージ）
            │                                         │
            │                                    全ステージクリア
            │                                         │
            │                                       Ending
            │
            └── 穴に落下 ──→  GameOver ──→  Playing（同ステージ リトライ）
```

### 描画構成

| 描画要素 | DrawCommand | 備考 |
|:---|:---|:---|
| 空 | `Skybox` | SimpleBox3D と同じ |
| フロア（通常タイル） | `Box3D`（グレー） | 薄い板として並べる |
| フロア（穴） | 描画なし | 穴は「タイルがない場所」で表現 |
| ボール | `Box3D`（白） | 0.8×0.8×0.8 |
| ゴール | `Box3D`（緑、半透明） | 1.0×0.2×1.0 の薄い板 |
| 障害物 | `Box3D`（赤） | 1.0×1.0×1.0 |
| ステージ番号 | egui HUD | |
| 残りリトライ数 | egui HUD | 最大 3 回 |

### コンポーネント構成

```
GameContent.RollingBall
  ├── SpawnComponent     # ステージ定義の読み込み・フロアタイル生成
  ├── PhysicsComponent   # ボール速度・慣性・穴落下判定（Elixir 側で近似）
  ├── CollisionComponent # 障害物・ゴールとの衝突判定
  └── RenderComponent    # DrawCommand 組み立て・push_render_frame
```

## 期待する発見

このコンテンツを作ることで以下の課題が具体的に現れると予想する：

1. **球メッシュの不在**
   現在の `DrawCommand` に球・円柱がない。`Box3D` で代替するか、
   新しい DrawCommand（`Sphere3D` 等）を追加するかの判断が必要になる。

2. **Elixir 側物理近似の限界**
   慣性・摩擦・弾き返しを Elixir で毎フレーム計算すると、
   `on_nif_sync` の処理時間が増える。どこまで Elixir で近似できるかが問われる。

3. **タイル単位のフロア描画**
   10×10 のフロアを `Box3D` で表現すると 100 個の DrawCommand が必要になる。
   現在の `Box3D` 上限（256 個）で足りるか、またバッチ描画の最適化が必要かが明らかになる。

4. **ステージデータの置き場所**
   フロアのタイル配置・穴の位置・障害物の座標をどこで定義するか。
   `game_content` 内に静的データとして持つか、外部ファイルから読み込むかの設計が問われる。

5. **新しいシーン（StageClear / Ending）**
   既存コンテンツにない「ステージクリア」「エンディング」シーンを追加することで、
   SceneManager のスタック管理が想定通りに機能するかが確認できる。
