## 1. Single Source of Truth（SSOT）の三層

マルチプレイで一貫性を説明するとき、次の三つに分ける。


| 層                   | 責務                           | 主な担い手                                          |
| ------------------- | ---------------------------- | ---------------------------------------------- |
| **状態の SSOT**        | 公式の世界状態（確定したエンティティ・ルール・スコア等） | **Elixir サーバー**                                |
| **入力のキャプチャ／ローカル意図** | デバイスからの取得・正規化・送信の起点          | **Rust クライアント**                                |
| **公式に適用される入力列**     | 順序・採否・検証後にシミュレーションに入るコマンド列   | **サーバー側で確定**（Elixir、または **Elixir が信頼する一本の経路**） |


クライアントは信頼できないため、「クライアントが送った内容＝無条件に真実」とはしない。**採用された入力の順序**はサーバーが宣言する。

**確定フレーム**（公式にコミットするタイムステップの区切り）は **Elixir** が行う。Rust 側で高頻度の計算や Zenoh 経由の入力反映を行う場合でも、**権威ある状態のコミット**は Elixir に収束させる。

## 2. 更新レートの区分（「何の Hz か」を混同しない）

### 2.1 クライアント描画フレームレート

- **ユーザー設定**可能にする。
- **デフォルト**は端末スペックに応じて **30 FPS または 60 FPS** を推奨。
- **上限**は利用ディスプレイ（**VR では HMD の表示モード**を含む）のリフレッシュレートに合わせる。

描画 Hz は体験品質の話であり、サーバー権威 tick と独立してよい。

### 2.2 シミュレーション（Rust / NIF 等）

- 可能なら **60 Hz** で更新したい。
- 60 Hz を超える負荷になる計算は **要件を緩める／間引く／別レーン**などで個別に扱う。

※ サーバー内 Rust の現状責務は [overview.md](./overview.md) を参照。本項は **目標方針**。

### 2.3 権威サーバー（Elixir）の tick

Elixir は計算集約の高頻度ループには向かないため、**公式状態の進行 tick** は次のように段階を設ける。


| モード | Hz     | 用途                                        |
| --- | ------ | ----------------------------------------- |
| 低速  | **10** | テスト・プロファイリング（Elixir／Rust 各モジュールのボトルネック特定） |
| 推奨  | **20** | 通常運用のデフォルト想定                              |
| 高速  | **30** | 遅延低減が必要な場合                                |


**Elixir を 60 Hz に近づけることは非推奨**（オーバーヘッド懸念）。

ネットワーク上でクライアントが「公式」とみなす状態更新の周期は、原則としてこの **Elixir tick** に引きずられる。Rust 側を 60 Hz で回す場合は、その結果を **いつ・どの形式で Elixir が取り込みコミットするか** をプロトコルで固定する（二重の真実を作らない）。

### 2.4 ネットワーク

- **低遅延**を優先。Zenoh・protobuf（`render_frame.proto` 等）の利用はその一環。
- プロバイダ契約 Mbps だけでなく、**アプリ実測**（送受バイト／秒、損失、RTT）を取得・記録し、品質プリセットや UI に反映する。**ユーザーごとの目標帯域を保証する**のではなく、**計測と把握**を主とする。

## 3. 可視範囲・同時可視（インタレスト管理）

- **距離カット・同時可視人数の上限**などは、**クライアントが調整**可能にする（PC スペック依存が大きいため）。
- **デフォルト**はクライアントがハードウェアを検出し、**自動推奨**する。

将来、不正や過負荷対策として **サーバー側のハードキャップ**を設けるかは別途検討可能。

---

## 4. 権限（ロール）と権威

権限階層の例:

`admin` > `builder` > `moderator` > `guest` > `spectator`

- **権威の複製（P2P で各自が正）**ではなく、**サーバー上のポリシー（RBAC）**として実装する。
- ロールは **サーバー側で毎回検証**する。クライアントの主張は信じない。
- `builder` 等の「権威」は、**サーバーがその操作を受理し状態を更新する**能力の意味合い。

---

## 5. リモートコンテンツの配布

- **改ざん・盗聴・なりすまし**を優先し、当面は **HTTPS ＋ 署名**（マニフェスト／ハッシュと作者または Hub の署名）を妥当な第一選択とする。
- 将来、転送経路だけ P2P 化する場合も、**同一の信頼の鎖（署名・コンテンツアドレス）** と両立させる。

---

## 6. 同時接続・パフォーマンス保証について

- **公式にサポートする同時人数・更新品質**は、**ホストマシンの CPU／メモリ**およびネットワーク条件に依存する。
- **ベンチマークとレポート**を経て数値目標を宣言する。

参考としての個人目標（未検証）:

- 既存サービスが許容するインスタンス規模（例: 数十人規模）を超える **同一空間の同時存在人数**（例として **255 人**）を視野に入れる。
- ただし **「全員フル同期・全員が互いにフル解像度」** と **描画 Hz／シム Hz／ネット更新 Hz** は別問題であり、**興味管理・間引き・LOD** とセットで定義する。

詳細なスケール上のギャップは [scale-and-gaps.md](../policy-as-code/gaps/scale-and-gaps.md) を参照。
