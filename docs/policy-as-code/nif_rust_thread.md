# Policy: NIF × Rust スレッド — 責務分担・Dirty NIF・委譲

[← index](./index.md)

---

## 1. 重い処理は Rust の独自スレッドで行う

**やってはいけないこと**: 式の演算（物理オーソリティ）・大量データ処理を NIF 内（スケジューラスレッド上）で同期的に実行すること。

**理由**:

- NIF は BEAM のスケジューラスレッドを占有する
- 重い処理を NIF に載せると、VM 全体の応答性が低下する
- 本プロジェクトの設計: NIF は「制御・データ注入」。サーバー側の実処理（式の演算・音の同期）は Rust の専用スレッドが担当する

**やるべきこと**: サーバー（NIF × Rust スレッド）は **式の演算**（物理オーソリティ）と **全クライアントの音の同期**（いつ何を再生するかの指示配信）を Rust の専用スレッドで行う。NIF は起動・パラメータ注入・イベント送受信など軽量な操作のみ行う。

---

## 2. 長時間 NIF が必要な場合は Dirty NIF または委譲を検討する

**やってはいけないこと**: 1ms 超の処理を通常 NIF で同期的に実行すること。

**やるべきこと**:

- **Dirty NIF**: `#[rustler::nif(schedule = "DirtyCpu")]` を用い、専用スレッドで実行させる。ただし dirty スレッド数には上限がある
- **委譲**: 処理を Rust の非同期タスクやチャネル経由で別スレッドに渡し、NIF は即座に返す。結果が必要な場合は Elixir へメッセージで非同期に返す

---

## 3. NIF と Rust ループの境界を明確にする

**やってはいけないこと**: Elixir から NIF を高頻度で呼び出して物理ステップを進めること。

**やるべきこと**: 物理ステップ（式の演算）はサーバー側 Rust の 60Hz ループで自律的に進め、NIF は「パラメータ更新」「イベント投入」「スナップショット取得」など、ループに影響を与える制御のみ行う。予測・補間はクライアント側で行う（[rust_client.md](./rust_client.md)）。
