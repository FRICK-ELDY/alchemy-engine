# Policy: Elixir — 責務・禁止事項

[← index](./index.md)

> **Elixir の得意分野**: Actor モデルによる並行処理・メッセージパッシング・障害耐性。軽量プロセスを大量に扱い、分散システムや I/O バウンドな処理に向く。演算には不向き。
>
> **補足（I/O バウンド）**: 入出力待ちで処理時間の大半が消費される状態。ネットワーク・DB・ディスクなどへの待ちが主となる。CPU バウンド（演算に時間がかかる）とは対照的で、Elixir は前者に適し、後者は Rust / Formula VM に向く。

> **2026-04 更新**: メインゲームループは **Elixir（contents）** がタイマー駆動（[architecture/overview.md](../architecture/overview.md)）。旧「サーバー Rust 60Hz `GameWorld` ループ」はない。下の数値は **過去の目安／負荷試験用** の参考として残す。

---

## 1. Elixir の刻みは「ハードリアルタイムの 60Hz 保証」ではない

**やってはいけないこと**: BEAM 上で **厳密な** 60Hz（約 16.67ms 毎）の**ハードリアルタイム**を保証できると仮定すること。

**理由**:

- メッセージ配送・スケジューリング・GC の揺らぎがあり、フレーム間隔は一定にならない
- 多数ルーム・多数プレイヤー時に、過度に高い刻みを要求するとキュー詰まりやスケジューラ圧迫のリスクが高まる
- クライアントの描画（約 60fps）とは別物。同期は protobuf / イベントで行う

**やるべきこと**: コンテンツが必要とするタイマー間隔を選び、遅延やスキップを設計に組み込む（バックプレッチャー等）。数式の重い部分は **Formula VM**（`run_formula_bytecode`）へ。クライアント側の補間・補外で見た目を滑らかにする（[rust_client.md](./rust_client.md)）。

| レート（参考・過去の目安） | 用途 |
|:---|:---|
| 10Hz | デバッグ・低負荷 |
| 20Hz | 安定運用のバランス例 |
| 30Hz | 上限に近い頻度の例（負荷・揺らぎに注意） |

## 2. 責務の逆転をしない（Elixir は SSoT）

**やってはいけないこと**: 責務を逆転させること。すなわち、Elixir が「実行」を担ったり、Rust が「定義」を持ったりすること。

**理由**:

- **Elixir**: **ドメイン**における定義の SSoT（Single Source of Truth）。「何を・どうするか」を決める責務を持つ（権威ある状態・ルール・コンテンツ）
- **`.proto`**: **Protobuf を使うペイロード**について、Elixir と Rust が同じバイト列で合意する **フィールド契約**の SSoT。UDP 外枠や Phoenix JSON など **他形式**は別の SSoT（[architecture/overview.md](../architecture/overview.md#設計思想)）。ゲームルールの SSoT に置き換わるものではない
- **Rust**: Elixir のドメイン定義を受け取り、それを効率よく「実行」する責務を持つ（クライアントは Protobuf 経路で `.proto` に従ってデコードする）
- 責務が逆転すると、ヘッドレスサーバーへの転用やコンテンツの差し替えが困難になり、アーキテクチャの一貫性が崩れる

**やるべきこと**: Elixir で定義し、NIF や Zenoh 経由で Rust（サーバー・クライアント両方）に注入する。Rust は注入された定義に従って SoA/SIMD で実行する。

## 3. Elixir では演算を行わない

**やってはいけないこと**: 数値演算・行列計算・物理計算・ループ内の計算処理を Elixir で実装すること。

**理由**:

Elixir は Erlang VM（BEAM）上で動作するため、演算が他の言語と比べて著しく遅い。1回の演算でも以下のオーバーヘッドが発生する：

```
VM命令のディスパッチ → 型チェック → ヒープから値の取得 → 演算 → 新しい値をヒープに生成 → GC対象化
```

- **ヒープベース**: すべての値がイミュータブルでヒープ上に存在し、演算のたびに新規オブジェクトが生成される
- **型の動的解決**: 実行時に型をチェックするオーバーヘッドがある
- **GC 負荷**: 計算が増えるほどヒープ割り当てが増え、GC の頻度とコストが上昇する

**やるべきこと**: 演算はすべて **Rust** 側で行う。Elixir は「何を計算するか」の定義・制御・オーケストレーションに徹する。
