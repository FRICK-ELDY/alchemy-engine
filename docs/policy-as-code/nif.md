# Policy: NIF — ブロック・シリアライズ・呼び出し頻度

[← index](./index.md)

---

## 1. NIF で VM を長時間ブロックしない

**やってはいけないこと**: NIF 内で数ミリ秒以上の CPU 処理や I/O 待ちを行うこと。通常の NIF は BEAM のスケジューラスレッド上で実行されるため、NIF がブロックしている間、そのスレッド上の全プロセスが止まる。

**理由**:

- BEAM は協調的スケジューリングを使用する。NIF が戻るまでスケジューラは他のプロセスに切り替えられない
- ブロック時間が長いと、ライブネス（応答性）が著しく低下し、数千プロセスを扱う設計の利点が失われる
- 物理演算・描画・オーディオなど重い処理は Rust の独自スレッドで行い、NIF は軽量な制御・データ注入に留める

**やるべきこと**: NIF は **1ms 未満**で返す。長時間処理が必要な場合は [nif_rust_thread.md](./nif_rust_thread.md) を参照し、dirty NIF または Rust 側の非同期処理に委譲する。

---

## 2. NIF 境界で大きなバイナリをシリアライズしない

**やってはいけないこと**: Elixir ↔ Rust の通信に、フレームごとの大容量バイナリ（位置・速度の配列、メッシュデータ、音声バッファなど）をシリアライズして渡すこと。

**理由**:

- シリアライズ/デシリアライズは CPU 負荷が高く、60Hz の物理ループや描画ループと同期する設計ではスケールしない
- ヒープコピーやエンコーディング変換が発生し、メモリ帯域と GC に負荷をかける
- 本プロジェクトの設計思想「Zero NIF Serialization Overhead」に反する

**やるべきこと**: 識別子（リソース参照・エンティティID・テーブルインデックスなど）のみを NIF 経由で渡し、実データは Rust 側に保持する。定義の注入は起動時やシーン遷移時など、低頻度のタイミングに限定する。

---

## 3. Elixir のループ内で NIF を高頻度呼び出ししない

**やってはいけないこと**: `Enum.map` / `for` / `Stream` などで、要素ごとに NIF を呼び出すこと。例: 敵 1000 体の位置更新を 1000 回の NIF 呼び出しで行う。

**理由**:

- NIF 呼び出しには境界越えのコストがある
- ループ回数分の呼び出しは、VM のスケジューリングとコンテキストスイッチのオーバーヘッドを増幅する
- バッチ処理（一括注入・一括取得）にすることで、呼び出し回数を削減できる

**やるべきこと**: データはまとめて渡し、NIF は 1 回または少数回で完了させる。Rust 側でループ・ベクトル化を行う。
