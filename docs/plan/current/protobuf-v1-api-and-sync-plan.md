# Protocol Buffers v1 — API・同期の段階導入 実装プラン

> 作成日: 2026-03-24  
> 目的: ワイヤ形式から Erlang term 依存を段階的に減らし、**まず Protocol Buffers（以下 Protobuf）だけ**で API とリアルタイム同期の **v1 契約**を切る。Cap’n Proto は v1 では採用せず、計測でボトルネックが確認されたメッセージ種のみ後から検討する。

---

## 1. 要約


| 項目              | 方針                                                                                                                                                                                                                                                                   |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **v1 の直列化**     | Protobuf のみ（`.proto` でスキーマを固定）                                                                                                                                                                                                                                       |
| **Cap’n Proto** | v1 では導入しない。ホットパスで CPU/レイテンシが不足した場合に限り、対象メッセージ種を個別検討                                                                                                                                                                                                                  |
| **既存ドキュメント**    | [network-protocol-current.md](../../architecture/network-protocol-current.md)、[erlang-term-schema.md](../../architecture/erlang-term-schema.md)、[zenoh-frame-serialization.md](../../policy-as-code/why_adopted/zenoh-frame-serialization.md) は、移行完了に合わせて更新または廃止誘導する |

## 2. 背景と目標

### 2.1 背景

- UDP の `:frame` 等で `term_to_binary` + zlib が使われている（[Network.UDP.Protocol](../../../apps/network/lib/network/udp/protocol.ex)）。
- Zenoh 経路では ETF（Erlang term）が採用されている（[erlang-term-schema.md](../../architecture/erlang-term-schema.md)）。
- Erlang term は BEAM 上では速い一方、**スキーマの明示的な進化**や**多言語クライアントとの契約**では不利になりやすい。

### 2.2 v1 で達成したいこと

1. **単一のスキーマ言語**（`.proto`）で、サーバー・クライアント（Elixir / Rust 等）が同じ型定義からコード生成できる。
2. **API（メタデータ・制御面）と同期ペイロード**を、まずはすべて Protobuf で統一し、運用・レビュー・CI の複雑さを抑える。
3. **後からの最適化余地**: 特定メッセージだけ別形式に差し替える判断を、計測結果に基づいて行えるようにする。

---

## 3. スコープ（v1）

### 3.1 含める


| 領域                   | 内容（例）                                                                               |
| -------------------- | ----------------------------------------------------------------------------------- |
| **ツールチェーン**          | `protoc` または [Buf](https://buf.build/) による lint/breaking 検知、生成物の配置規約                |
| **Elixir**           | 生成モジュールの置き場（例: `apps/network/lib/network/proto/` または専用アプリ）、Mix タスクまたはスクリプト          |
| **Rust（native）**     | `prost` / `tonic` 等の選定、ワークスペースでの生成パス                                                |
| **UDP 同期**           | `Network.UDP.Protocol` の `:frame` ペイロードを Protobuf メッセージに置き換え（圧縮方針は別途決定）             |
| **Channel（Phoenix）** | 現行の JSON ペイロードのうち、契約として固定したいものを Protobuf に移行する方針の確定と実装（バイナリフレーム、Base64 ラップ等は実装時に選択） |


### 3.2 v1 では含めない（明示的な非目標）

- **Cap’n Proto** の導入、または Protobuf と併用する二重スタック。
- **フル gRPC 化**の強制（必要なら後続フェーズとして切り出す。v1 は「メッセージ定義とエンコードの統一」が主目的）。
- **Zenoh レンダーフレーム（ETF）の一括置換**（データ量・パスが大きいため、UDP / Channel で手順を固めた後のフェーズとして扱うのが安全。§5.3 参照）。

## 5. Cap’n Proto を検討する条件（v1 完了後）

以下を**計測で**満たす場合に限り、対象を **メッセージ種単位** で検討する。プロジェクト全体の一括移行はしない。


| 指標         | 例                                                      |
| ---------- | ------------------------------------------------------ |
| CPU プロファイル | サーバーまたはクライアントで、Protobuf の encode/decode がホットスポットになっている |
| レイテンシ      | 同期フレームの p99 が要件を超え、ボトルネックがシリアライズに特定できる                 |
| ペイロード特性    | 同一レイアウトの繰り返し読み取りでゼロコピーが明確に有利                           |


判断時は [bottleneck-prevention.md](../../policy/bottleneck-prevention.md) に沿って記録する。

## 8. 実施順序の推奨

1. **フェーズ A** でツールチェーンと生成物の置き場を固める。
2. **フェーズ B** で UDP の `:frame` を v1 の中心として置き換え（影響範囲が読みやすい）。
3. **フェーズ C** で Channel 等の API 面を追従。
4. **フェーズ D** はデータ量が大きいため、B/C の運用知見を踏まえて着手。
5. 計測のうえ、必要なら **Cap’n Proto を特定メッセージのみ**検討。

## 9. チェックリスト（プラン完了の定義）

本プラン文書としての「初版完了」は、上記フェーズ A〜C のタスクが **Issue / PR に分解可能な粒度**で表に落ちている状態とする。実装完了の定義は各フェーズの PR ごとに別途記載する。

- `proto/`（または決定した配置）と命名規約の README または本書への追記  
- フェーズ A のスモーク（生成パイプライン）が再現可能  
- フェーズ B の `.proto` 草案と既存イベント型の対応表（ドラフトで可）  
- フェーズ C の Channel イベント一覧と `.proto` の対応方針  
- フェーズ D を「後続」と明記し、依存関係に矛盾がないこと
