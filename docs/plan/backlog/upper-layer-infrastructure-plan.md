# 上位レイヤー基盤整備計画書

> 認証・インスタンスレジストリ・ディスカバリを段階的に整備するための計画書。
> 最終ゴールである「分散型VR空間」（`vision-goal.md`）と、
> 「ログアウト後ヘッドレス継続・インスタンス譲渡」のビジョン実現に向けた基盤となる。

---

## 1. 計画の背景と目的

### 1.1 達成したいこと

| 目標 | 説明 |
|:---|:---|
| **ヘッドレス継続** | ユーザーがログアウトしてもヘッドレスサーバーは動き続け、インスタンス内の参加者は遊び続けられる |
| **再ログイン時の復帰** | ログインしたユーザーが、前回いたインスタンスに再参加できる |
| **PC起動時バックグラウンド起動** | PC起動時にヘッドレスサーバーをバックグラウンドで起動し、ユーザー登録済みインスタンスを自動起動 |
| **ヘッドレスのみ運用** | 描画なしでサーバーのみ稼働するモードの確立 |
| **インスタンス譲渡** | ホストPCが落ちる際、他ユーザーのヘッドレスにインスタンスを委譲できる（VRChat マスター譲渡 + NeosVR/Resonite インスタンス移行のハイブリッド） |

### 1.2 本計画で整備する上位レイヤー

以下3つの基盤を整備することで、上記の目標を支える：

- **認証** — ユーザーアイデンティティ・ログイン/ログアウト・セッション管理
- **インスタンスレジストリ** — ユーザーが登録したインスタンスの永続化・起動設定の管理
- **ディスカバリ** — どのインスタンスがどのノード（どのPC）で動いているかの検出・一覧

### 1.3 既存ドキュメントとの関係

- `docs/vision-goal.md` — 分散型VR SNS のビジョン。Phase 3〜5 の前提となる
- `docs/vision.md` — エンジン単体の保証範囲
- 本計画は vision-goal の Phase 3（単一インスタンスのネットワーク公開）以降と整合する

## 2. 認証レイヤー

### 2.1 責務

| 責務 | 説明 |
|:---|:---|
| ユーザーアイデンティティ | `@user@alchemy.example.com` 形式の識別子 |
| ログイン/ログアウト | 認証状態の確立と破棄。**ログアウト時もインスタンス（Room）プロセスは停止しない** |
| セッション管理 | トークン・セッションID による接続紐づけ |
| オーナーシップ | インスタンスとオーナー（ユーザー）の対応。譲渡時に必要 |

### 2.2 設計上の分離

```
ユーザーセッション層  ─── ログイン/ログアウトの対象
        │
        │ 独立
        ▼
Room（インスタンス）層  ─── 物理・ゲームロジック。セッション有無に依存しない
```

- Room は「誰がオーナーか」を知るが、「誰がログイン中か」に生存を依存しない
- ログアウト = クライアント接続の切断 + 認証状態の破棄のみ

### 2.3 検討事項（未決定）

| 項目 | 候補 | 備考 |
|:---|:---|:---|
| 認証方式 | OAuth2/OIDC / 独自 / 匿名可 | フェデレーション時は OIDC が有力 |
| ホームインスタンス | ユーザー登録先インスタンス | vision-goal の `@user@instance` モデル |
| セッション永続化 | JWT / サーバーセッション | クロスインスタンス入室時は JWT が扱いやすい |

### 2.4 成果物（想定）

- 認証モジュール（`apps/game_auth` 等）
- ユーザー・セッションのスキーマ（DB または ETS/永続化）
- ログイン/ログアウト API（Phoenix / UDP ゲートウェイ経由）

## 3. インスタンスレジストリ

### 3.1 責務

| 責務 | 説明 |
|:---|:---|
| 登録インスタンスの永続化 | ユーザーが「このインスタンスを自分のものとして運用する」と登録した一覧 |
| 起動設定の保存 | コンテンツ種別・マップ・初期パラメータなど |
| PC起動時の復元 | アプリ起動時に登録済みインスタンスを `RoomSupervisor.start_room/1` で起動 |

### 3.2 データモデル（案）

```elixir
# 登録インスタンス（永続化単位）
%RegisteredInstance{
  id: String.t(),           # UUID
  owner_user_id: String.t(), # オーナーのユーザーID（認証レイヤーと紐づく）
  room_id: atom() | String.t(),
  content_module: module(), # GameContent.VampireSurvivor 等
  map_id: atom(),
  created_at: DateTime.t(),
  updated_at: DateTime.t()
}
```

### 3.3 起動フロー（想定）

```
PC起動
  → GameServer.Application.start/2
  → InstanceRegistry.load_registered/0  # 永続化から読み込み
  → Enum.each(instances, &RoomSupervisor.start_room/1)
  → 各 Room がヘッドレスで稼働開始
```

### 3.4 検討事項（未決定）

| 項目 | 候補 | 備考 |
|:---|:---|:---|
| 永続化先 | SQLite / ETS + ファイル / PostgreSQL | セルフホスト重視なら SQLite が現実的 |
| 1ユーザーあたりの上限 | 未定 | リソース制限と UX のバランス |
| インスタンスIDの生成 | UUID / ユーザー指定名 | 譲渡・移行時に一意性が重要 |

### 3.5 成果物（想定）

- インスタンスレジストリモジュール（`InstanceRegistry`）
- 永続化スキーマ・マイグレーション
- `Application.start/2` での登録インスタンス自動起動フック

## 4. ディスカバリ

### 4.1 責務

| 責務 | 説明 |
|:---|:---|
| ノードの検出 | どの PC（BEAM ノード）が稼働中か |
| インスタンスの所在 | どのインスタンスがどのノードで動いているか |
| 譲渡先候補の取得 | インスタンス移行・譲渡時に「受け入れ可能なノード」の一覧 |

### 4.2 スコープ

- **ローカル（同一マシン内）**: 現状の `RoomRegistry` で十分
- **フェデレーション（他インスタンス・他ノード）**: プロトコルとトポロジーが必要

本計画では、**同一マシン内** と **同一 LAN / 信頼できる少数ノード** を第一段階とする。

### 4.3 設計案（段階的）

| 段階 | 範囲 | 方式 |
|:---|:---|:---|
| Phase A | 単一ノード | `RoomRegistry` + `InstanceRegistry` の組み合わせで十分 |
| Phase B | 同一ユーザーの複数デバイス | ユーザーIDで紐づけ、ディスカバリは手動登録 |
| Phase C | フェデレーション | インスタンス間プロトコル（ActivityPub 的 または 独自）で相互発見 |

### 4.4 検討事項（未決定）

| 項目 | 候補 | 備考 |
|:---|:---|:---|
|  discovery プロトコル | mDNS / 手動登録 / 中央レジストリ | セルフホストでは mDNS が扱いやすい |
| ノード間通信 | libcluster / 直接 TCP | Elixir なら libcluster が一般的 |
| 譲渡先選定ロジック | ラウンドロビン / 負荷ベース / 参加者に近いノード | 将来の最適化 |

### 4.5 成果物（想定）

- ディスカバリモジュール（`InstanceDiscovery`）
- ノード一覧・インスタンス所在テーブル（メモリ or 分散 ETS）
- 譲渡リクエスト時の「受け入れ可能ノード」取得 API

## 5. 実装フェーズ

### Phase 1: 基盤の土台（認証なし・ローカルのみ）

| タスク | 説明 | 依存 |
|:---|:---|:---|
| インスタンスレジストリ（永続化） | 登録インスタンスの保存・読み込み | なし |
| 起動時登録インスタンス自動起動 | `Application.start` で `InstanceRegistry` から起動 | 上記 |
| ヘッドレス専用起動モード | 描画・ウィンドウを完全に無効化したモード | 既存 headless フィーチャー |

**成果**: PC起動時に登録済みインスタンスが自動で立ち上がり、ヘッドレスのみで稼働できる。

### Phase 2: 認証の導入

| タスク | 説明 | 依存 |
|:---|:---|:---|
| ユーザーアイデンティティ | `@user@instance` 形式の識別 | Phase 1 |
| ログイン/ログアウト API | 認証状態の確立・破棄 | 上記 |
| セッションと Room の分離 | ログアウト時も Room は生存 | 上記 |
| インスタンスオーナーシップ | 登録インスタンスとユーザーの紐づけ | Phase 1, 認証 |

**成果**: ユーザーがログアウトしてもインスタンスは動き続け、再ログイン時に同じインスタンスに戻れる。

### Phase 3: ディスカバリと譲渡の基盤

| タスク | 説明 | 依存 |
|:---|:---|:---|
| 単一ノード内ディスカバリ | `RoomRegistry` と連携したインスタンス一覧 API | Phase 2 |
| マスター譲渡（権限の移譲） | オーナーシップの譲渡 API | Phase 2 |
| 譲渡リクエストフロー | 譲渡希望 → 譲渡先選定 → オーナー更新 | 上記 |

**成果**: ホストが抜ける際に、別ユーザーにオーナー権限を譲渡できる（VRChat 的）。

### Phase 4: インスタンス移行（将来）

| タスク | 説明 | 依存 |
|:---|:---|:---|
| 状態スナップショットのネットワーク転送 | `SaveManager` 形式を拡張し、他ノードへ送信可能に | Phase 3 |
| 移行先での復元 | 受信したスナップショットから `GameEvents` を起動 | 上記 |
| クライアント再接続 | 移行先のエンドポイントへ接続し直す | 上記 |

**成果**: ホスト PC が落ちる前に、インスタンス全体を他ノードへ移行できる（NeosVR/Resonite 的）。
