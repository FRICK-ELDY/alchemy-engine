# BasisVR / Basis — 競合・参照プロジェクトとしての位置づけ

## 参照先

- 上流リポジトリ: [BasisVR/Basis](https://github.com/BasisVR/basis)（MIT、Unity 6 + C#）
- ローカル解析用クローン例: `C:\Users\ncssa\github\BasisVR\Basis`（チーム内でパスは任意）

## このフォルダの目的

AlchemyEngine と **思想的に近い**（自前ホスト可能なソーシャル VR / ネットワーク化の土台を OSS で提供する）一方で、**実装スタックは根本から異なる**（Unity クライアント + C# サーバ、UDP/LiteNetLib 中心）競合・先行事例として Basis を追跡する。

ここでやることは次の通りに限定する。

1. **意図の共有**: 何を学び、何を踏襲しないかをチーム内で揃える（本ファイル）。
2. **構造の把握**: コードベース上のモジュール境界とデータの流れを短く固定する（[architecture.md](./architecture.md)）。
3. **設計への入力**: Alchemy のプロトコル・スケール・運用方針の議論に、根拠付きで材料を出す（数値ベンチはコード・公式投稿など一次情報と照合する。例: [負荷テスト動画（X）](https://x.com/BasisVR/status/2044403555805827320)、[オーディオ劣化・対策の言及（X）](https://x.com/BasisVR/status/2000152570523009327)）。

## Alchemy との関係（要約）

| 観点 | Basis（把握している範囲） | AlchemyEngine（本リポジトリ） |
|------|---------------------------|-------------------------------|
| 製品形 | Unity デモ + ライブラリ群、ヘッドレス `Basis Server` | Elixir / Rust NIF / Zenoh 等を軸にしたエンジン + コンテンツ分離 |
| 同期・帯域 | サーバ上で距離別アバター品質・送信間隔の削減（BSR）、チャネル分割が明示的 | 別ドキュメント群で定義（Zenoh、権威ある状態同期ポリシー等） |
| ボイス／オーディオ | 受信者リストの複数表現、Opus 等（リポジトリ README の依存）。公開情報では **約 300〜400 接続付近でオーディオ劣化**が報告され改善に注力との旨（[X](https://x.com/BasisVR/status/2000152570523009327)） | [音声の責務分担](../../policy/audio-responsibility.md) を起点に、大人数時の劣化対策を設計で追う必要あり |
| 哲学 | 「クリエイターの代理店にならない」に近い土台提供（[PHILOSOPHY](https://github.com/BasisVR/Basis/blob/developer/PHILOSOPHY.md)） | [docs/vision.md](../../vision.md) で「器」とコンテンツを厳密に分離 |

## 解析で狙うこと（良いところの模倣候補）

- **スケール志向がコードに落ちている**こと（例: パケットプールサイズや「1000 接続」を前提にしたコメント、距離キャッシュでホットパスから float 演算を外す等）。詳細は [architecture.md](./architecture.md)。
- **関心の分離**: チャネル番号とメッセージ型、還元システム（BSR）、認証・権限・モデレーションがファイル単位で分かれている点。
- **運用**: Docker、`config.xml` と初期リソースディレクトリ、コンソールコマンドなど自前ホスト向けの導線。
- **オーディオのスケール実感**: 先行事例として「数百人でボイス品質が先に効く」可能性を認識し、Alchemy でも帯域・ミキング・受信者絞り込みを早めに検証する（[architecture.md](./architecture.md) §11.2）。

## 解析で避けたい落とし穴（単純コピーしない）

- **スタック前提の最適化**（.NET 単一プロセス + LiteNetLib + Unity シリアライズ）を、そのまま Alchemy の Zenoh / Elixir モデルに当てはめない。
- **「1 インスタンス 1000 人」等の数字**は実験条件（アバター複雑度、ボイス有無、同時発話率、地理分散）に依存する。公開されている説明は動画ベースのため、数値を設計根拠にするときは条件の言語化が必要（参照: [BasisVR の負荷テスト投稿（X）](https://x.com/BasisVR/status/2044403555805827320)）。クライアントは Unity レンダリングを含むため、**サーバ・ネットワーク単体の上限ではない**点に注意する。
- **クライアントとサーバのコード共有**（Unity パッケージ内にサーバ DLL 相当が同居）は開発速度には効くが、Alchemy の言語分割方針とはトレードオフが異なる。
